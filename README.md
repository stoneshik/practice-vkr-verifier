# DOCX Verifier (ВКР)

Модуль для автоматической проверки выпускных квалификационных работ (ВКР) в формате `.docx`.

Проверяет:
- структуру документа (разделы, оглавление)
- форматирование (шрифт, интервалы, поля)
- оформление заголовков
- рисунки и таблицы
- формулы
- списки, термины, сокращения
- список источников

Результат - JSON-отчёт с ошибками, предупреждениями и рекомендациями.

Используется Python 3.10

## Назначение модулей

### Core

- `cli.py` - точка входа CLI, обработка аргументов
- `model.py` - построение внутренней модели документа
- `docx_reader.py` - чтение DOCX на уровне XML
- `pagination.py` - вычисление страниц
- `report.py` - формирование итогового отчёта

### Utilities

- `constants.py` - константы и регулярные выражения
- `utils.py` - вспомогательные функции

- `file_checks.py` - проверка файла
- `structure_checks.py` - структура документа
- `formatting_checks.py` - форматирование
- `objects_checks.py` - таблицы и рисунки
- `special_checks.py` - формулы, списки, источники

## Запуск

Запускать нужно из родительской директории
```bash
./verifier/.venv/bin/python -m verifier --pretty ./verifier/uploads/ideal.docx
```

Также в директории ./uploads расположены примеры файлов для тестов.
* `first.docx` - файл в целом правильным, но с небольшим количеством ошибок и замечаний
* `ideal.docx` - за основу взят файл `first.docx`, но все ошибки и замечания исправлены
* `much_errors.docx` - за основу взят файл `first.docx`, но намерено испорчен и содержит большое количество ошибок
* `first.pdf` - файл `first.docx` преобразован в формат в pdf при попытке анализа выдает ошибку

## Использование в системе (Spring Boot)

Модуль используется как CLI-инструмент.

Сценарий:

1. Пользователь загружает .docx  
2. Spring сохраняет файл  
3. Spring запускает Python:
    ```bash
    python -m verifier /path/to/file.docx  
    ```
4. Получает JSON отчет из stdout  
5. Сохраняет результат в БД  
6. Возвращает результат клиенту

# Полная структура JSON-отчёта

Отчёт, возвращаемый модулем, имеет следующую структуру:

```json
{
  "status": "completed | completed_with_findings | error",
  "file": { ... },
  "statistics": { ... },
  "summary": { ... },
  "document": { ... },
  "findings": [ ... ],
  "meta": { ... }
}
```

---

## Поле `status`

Статус выполнения анализа:

* `completed` - ошибок и предупреждений нет
* `completed_with_findings` - есть ошибки или предупреждения
* `error` - произошла ошибка при обработке файла

---

## Поле `meta`

```json
"meta": {
  "analysis_time_ms": 120
}
```

Служебная информация:

* `analysis_time_ms` - время выполнения анализа

---

# Массив `findings`

Основной результат работы системы - список найденных нарушений.

```json
"findings": [
  {
    "rule": "F05",
    "severity": "error",
    "message": "Точка в конце номера заголовка глав не допускается",
    "location": {
      "paragraph": 27,
      "page": 7
    },
    "actual": "1.",
    "expected": "1",
    "recommendation": "Удалить точку после номера заголовка",
    "evidence": null
  }
]
```

---

## Поля объекта `finding`

### `rule`

Код правила.

Примеры:

* `F01` - проверка формата файла
* `F05` - заголовки
* `F07` - форматирование текста

Используется для:

* фильтрации
* аналитики
* отображения на UI

---

### `severity`

Уровень проблемы:

* `error` - критическая ошибка (нарушение требований)
* `warning` - нежелательное отклонение
* `info` - информационное сообщение

---

### `message`

Человекочитаемое описание проблемы.

---

### `location`

Локация проблемы в документе:

```json
"location": {
  "paragraph": 27,
  "block": 30,
  "section": 2,
  "page": 7
}
```

Поля могут отсутствовать.

* `paragraph` - номер абзаца (1-based)
* `block` - индекс блока (абзац/таблица)
* `section` - номер секции
* `page` - вычисленная страница

---

### `actual`

Фактическое значение.

Пример:

```json
"actual": "Times New Roman 12pt"
```

---

### `expected`

Ожидаемое значение.

```json
"expected": "Times New Roman 14pt"
```

---

### `recommendation`

Рекомендация по исправлению.

```json
"recommendation": "Установить кегль 14 pt"
```

---

### `evidence` (опционально)

Дополнительные данные для диагностики.

Может содержать:

* XML-фрагменты
* список значений
* сырой текст

---

# Особенности массива findings

## 1. Неполные поля

Не все поля обязательны:

* `location` может содержать только `paragraph`
* `page` добавляется отдельно после вычисления
* `actual/expected` могут отсутствовать

---

## 2. Порядок

Элементы идут в порядке обнаружения при анализе.

---

## 3. Группировка

Группировка выполняется через:

```json
"summary.by_rule"
```

---

## 4. Связь с document

* `findings` - результат проверки
* `document` - контекст

Пример:

* `findings` говорит: "ошибка в заголовке"
* `document.structure_overview` показывает, где этот заголовок находится

---

# Итог

* `findings` - основной результат (что не так)
* `summary` - агрегированная статистика
* `document` - контекст (почему и где)
* `file/statistics/meta` - служебная информация

Это разделение позволяет:

* удобно строить UI
* легко расширять проверки
* дебажить без повторного анализа DOCX

## Структура JSON-отчёта: вспомогательная часть

Помимо массива `findings`, отчёт содержит вспомогательные блоки, которые нужны для диагностики, отображения структуры документа и служебной информации.

---

## Блок `file`

Содержит метаданные проверяемого файла.

Пример:

```json
"file": {
  "name": "ideal.docx",
  "path": "/home/sus/Desktop/university/practice/verifier/uploads/ideal.docx",
  "size_bytes": 56072,
  "suffix": ".docx"
}
````

### Поля

* `name` - имя файла без изменений
* `path` - абсолютный путь к файлу на сервере
* `size_bytes` - размер файла в байтах
* `suffix` - расширение файла

### Для чего нужен

Этот блок используется для:

* идентификации файла в отчёте
* сохранения сведений о проверенном документе в БД
* последующего отображения пользователю

---

## Блок `statistics`

Содержит агрегированную статистику по документу.

Пример:

```json
"statistics": {
  "paragraphs": 64,
  "tables": 4,
  "inline_shapes": 1,
  "sections": 9,
  "finding_count": 6
}
```

### Поля

* `paragraphs` - количество абзацев в документе
* `tables` - количество таблиц Word
* `inline_shapes` - количество встроенных графических объектов
* `sections` - количество секций документа
* `finding_count` - общее количество найденных замечаний

### Для чего нужен

Этот блок нужен для:

* краткой технической сводки по документу
* контроля полноты парсинга
* отображения статистики в интерфейсе

---

## Блок `summary`

Содержит краткую сводку по результатам проверки.

Пример:

```json
"summary": {
  "errors": 0,
  "warnings": 4,
  "info": 2,
  "by_rule": {
    "F10": 1,
    "F14": 1,
    "F17": 4
  }
}
```

### Поля

* `errors` - количество ошибок
* `warnings` - количество предупреждений
* `info` - количество информационных сообщений
* `by_rule` - распределение замечаний по правилам

### Для чего нужен

Этот блок нужен для:

* быстрого понимания результата проверки
* построения сводки на фронтенде
* фильтрации и аналитики по типам нарушений

---

## Блок `document`

Содержит дополнительную структурную информацию о документе.
Этот блок не является списком ошибок, а даёт контекст для диагностики и отображения результатов.

Структура:

```json
"document": {
  "structure_overview": { ... },
  "sections": [ ... ],
  "raw_toc": [ ... ]
}
```

---

## Блок `document.structure_overview`

Содержит краткую структурную карту документа.

Пример:

```json
"structure_overview": {
  "headings": [
    {
      "paragraph": 1,
      "text": "СПИСОК СОКРАЩЕНИЙ И УСЛОВНЫХ ОБОЗНАЧЕНИЙ",
      "level": 1,
      "page_break_before": false,
      "has_page_break": false,
      "page": 4
    }
  ],
  "figure_paragraphs": [63],
  "tables": [1, 2, 3, 4]
}
```

### Поля

#### `headings`

Список найденных заголовков.

Пример элемента:

```json
{
  "paragraph": 16,
  "text": "ВВЕДЕНИЕ",
  "level": 1,
  "page_break_before": false,
  "has_page_break": false,
  "page": 6
}
```

Описание полей:

* `paragraph` - номер абзаца, в котором найден заголовок
* `text` - текст заголовка
* `level` - уровень заголовка
  * `1` - глава или крупный структурный раздел
  * `2` - подраздел
  * `3+` - более глубокий уровень вложенности
* `page_break_before` - установлен ли у абзаца признак разрыва страницы перед ним
* `has_page_break` - найден ли разрыв страницы в XML абзаца
* `page` - вычисленная страница, на которой расположен заголовок

#### `figure_paragraphs`

Список номеров абзацев, в которых обнаружены графические объекты.

Пример:

```json
"figure_paragraphs": [63]
```

#### `tables`

Список индексов таблиц в документе.

Пример:

```json
"tables": [1, 2, 3, 4]
```

### Для чего нужен

Блок `structure_overview` используется для:

* визуального представления структуры документа
* отладки алгоритмов поиска заголовков
* анализа расположения рисунков и таблиц
* проверки корректности оглавления и пагинации

---

## Блок `document.sections`

Содержит сведения о секциях Word.

Пример элемента:

```json
{
  "section_index": 0,
  "left_mm": 30.0,
  "right_mm": 14.99,
  "top_mm": 20.0,
  "bottom_mm": 20.0,
  "page_width_mm": 210.01,
  "page_height_mm": 297.0,
  "different_first_page": false,
  "odd_and_even_pages": false,
  "footer_alignment": null,
  "footer_has_page_field": true,
  "header_has_page_field": false,
  "header_has_text": false,
  "footer_has_text": false,
  "footer_xml_snippet": "<w:ftr ...>"
}
```

### Поля

* `section_index` - индекс секции, начиная с нуля
* `left_mm` - левое поле страницы в миллиметрах
* `right_mm` - правое поле страницы в миллиметрах
* `top_mm` - верхнее поле страницы в миллиметрах
* `bottom_mm` - нижнее поле страницы в миллиметрах
* `page_width_mm` - ширина страницы
* `page_height_mm` - высота страницы
* `different_first_page` - используется ли отдельный колонтитул для первой страницы
* `odd_and_even_pages` - включены ли разные колонтитулы для чётных и нечётных страниц
* `footer_alignment` - выравнивание текста в футере
* `footer_has_page_field` - найдено ли поле `PAGE` в нижнем колонтитуле
* `header_has_page_field` - найдено ли поле `PAGE` в верхнем колонтитуле
* `header_has_text` - есть ли текст в верхнем колонтитуле
* `footer_has_text` - есть ли текст в нижнем колонтитуле
* `footer_xml_snippet` - укороченный XML футера для диагностики

### Для чего нужен

Этот блок нужен для:

* проверки параметров страницы
* анализа настройки нумерации страниц
* диагностики колонтитулов
* поиска причин ошибок в правилах F08 и F10

---

## Блок `document.raw_toc`

Содержит сырые данные оглавления, извлечённые из DOCX.

Пример элемента:

```json
{
  "title": "ВВЕДЕНИЕ",
  "page": "6",
  "has_separator": true,
  "leader_type": "dot",
  "has_leader": true,
  "raw": "ВВЕДЕНИЕ\t6"
}
```

### Поля

* `title` - заголовок из строки оглавления
* `page` - номер страницы, извлечённый из строки
* `has_separator` - удалось ли подтвердить наличие разделителя между названием и номером страницы
* `leader_type` - тип лидера табуляции, например `dot`
* `has_leader` - удалось ли подтвердить наличие заполнителя
* `raw` - исходная строка оглавления в почти сыром виде

### Для чего нужен

Этот блок используется для:

* проверки правила F06
* сравнения оглавления с реальными заголовками документа
* анализа корректности заполнителей и табуляции
* отладки парсинга оглавления

### Полный список правил проверки

`F01`
Проверяет, что в систему загружен файл допустимого типа и что сам сценарий обработки DOCX вообще может быть запущен.

`F02`
Проверяет расширение файла и MIME-тип, чтобы отсечь неподходящие или поддельные файлы.

`F03`
Отвечает за извлечение данных из документа: абзацы, таблицы, рисунки, формулы, стили, поля страницы, колонтитулы и другую структуру.

`F04`
Проверяет наличие обязательных частей ВКР: содержание, введение, заключение, список источников, а также контролирует базовую корректность структурных заголовков.

`F05`
Проверяет правильность нумерации глав и подразделов, отсутствие точки после номера главы, а также начало крупных структурных разделов с новой страницы.

`F06`
Проверяет оглавление: есть ли оно, совпадают ли его пункты с заголовками документа, есть ли разделитель между названием и номером страницы, соответствуют ли страницы.

`F07`
Проверяет основной текст на соответствие требованиям: Times New Roman, 12–14 pt, чёрный цвет, интервал 1.5, абзацный отступ 1.25 см, выравнивание по ширине.

`F08`
Проверяет поля документа: левое 30 мм, правое 10–15 мм, верхнее 20 мм, нижнее 20 мм.

`F09`
Проверяет оформление заголовков: допустимый регистр, полужирность, отсутствие точки в конце, отсутствие аббревиатур, отсутствие переносов, наличие абзацного отступа.

`F10`
Проверяет наличие номера страницы, его расположение в центре нижнего поля и настройку отдельного колонтитула для первой страницы.

`F11`
Проверяет, что у рисунка есть подпись формата `Рисунок N — Название` и что она расположена под изображением по центру.

`F12`
Проверяет, что у таблицы есть подпись формата `Таблица N — Название`, расположенная над таблицей, и что таблица вставлена как объект Word.

`F13`
Проверяет приложения: наличие заголовка вида `ПРИЛОЖЕНИЕ А`, центрирование и начало каждого приложения с новой страницы.

`F14`
Проверяет наличие необязательных, но предусмотренных методичкой разделов: список сокращений, термины и определения, список иллюстративного материала.

`F15`
Проверяет оформление списка сокращений: элементы столбцом, расшифровка через тире, отсутствие точки и других завершающих знаков в конце строк.

`F16`
Проверяет раздел терминов: либо строки вида `Термин — определение`, либо таблицу из двух колонок `Термин` / `Определение`.

`F17`
Проверяет формулы: вынесены ли они в отдельную строку, есть ли пояснение со слова `где`, есть ли номер формулы, используются ли неразрывные пробелы между числом и единицей измерения.

`F18`
Проверяет наличие раздела со списком источников и корректную нумерацию источников арабскими цифрами.

`F19`
Проверяет списки и перечисления: единообразие маркеров, пунктуацию, вложенность и базовую согласованность оформления.

`F20`
Отвечает за сбор и выдачу итогового отчёта: тип нарушения, уровень серьёзности, местоположение, фактическое значение, ожидаемое значение, рекомендация.
